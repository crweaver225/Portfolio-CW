One of the most highly touted features of Haskell is its rich, strong type system. The flux in types we see in JavaScript and Python do not exist, and the implicit conversion from one type to another that we see in C/C++ is avoided as well. Haskell comes with its own set of primative types that can be leveraged to write interesting software, but often it is important for developers to be able to create their own types. Haskell has some really unique and interesting offerings for developers looking to build and utilize custom types in their programs.

Value Constructor

We will start by creating a value constructor, which is a special function that lets us create a new value of a certain type

Haskell
1data CarInformation = CarInformation String

We now have a new type called CarInformation with an String ‘inhabitant’

Haskell
1myCar :: CarInformation
2myCar = CarInformation "Honda Civic"

We can use our new type like so

Haskell
1showCar :: CarInformation -> String
2showCar (CarInformation carType) = 
3    "Car type: " <> carType

Our types can contain more than one value if we so desire

Haskell
1data CarInformation = CarInformation String String Int Int
2myCar :: CarInformation
3myCar = CarInformation "Honda" "Civic 2014 25

Our CarInformation now stores the car make, model, year, and MPG. In addition to the function above for building a function to access information, we can also utilize pattern matching

Haskell
1carMake :: CarInformation -> String
2carMake (CarInformation make _ _ _ ) = make
3
4carYear :: CarInformation -> Int
5carYear (CarInformation _ _ year _) = year

And updating an instance of a type (say if the MPG goes down with age)

Haskell
1updateMPG :: CarInformation -> CarInformation
2updateMPG (CarInformation make model year _ ) mpg =
3    CarInformation make model year mpg

Records

The immediate problem with what we have done so far is that we are required to know the order of values within a type when working with them. This can be challenging, particularly for types with lots of data fields within them. Luckily, Haskell has a special syntax for building types with named parameters. These are called Records

Haskell
 1data CarInformation = CarInformation
 2{   make :: String,
 3    model :: String,
 4    year :: Int,
 5    mpg :: Int
 6}
 7
 8mySecondCar :: CarInformation
 9mySecondCar = 
10    CarInformation
11    { make = "Dodge",
12      model = "Stratus",
13      year = 2016,
14      mpg = 18
15    }

Accessing these values because significantly more trivial and also allows us to compose really succinct functions

Haskell
1show $ year mySecondCar
2-- prints: 2016
3
4totalMpgCount :: [CarInformation] -> Int
5totalMpgCount = sum . map mpg

The second function can take a list of CarInformation types and sum up the miles per gallon.

Sum Types

The real value in Haskell comes from its algebraic data types, also referred to as sum types which allows us to represent between multiple different types within another type

Haskell
1data Direction = | North | East | South | West
2
3whichDirection :: Direction -> String
4whichDirection direction =
5    case direction of
6        North -> "North"
7        East -> "East"
8        South -> "South"
9        West -> "West"

We can leverage these sum types along with our data types to create powerful abstractions

Haskell
 1data PreferredContactMethod = Email String | TextMessage String | Mail String String String Int
 2
 3emailContact :: PreferredContactMethod
 4emailContact = Email "example@example.com"
 5
 6textContact :: PreferredContactMethod
 7textContact = TextMessage "111-111-1111"
 8
 9mailContact :: PreferredContactMethod
10mailContact = Mail "123 main st." "SmallVille" "Illinois" 60543

Similar to a variant in C++ or a union in C, this sum type allows us to generalize our work

Haskell
1confirmContact :: PreferredContactMethod -> String
2confirmContact contact =
3    case contact of
4        Email emailAddress -> "Ok, I will email you at " <> emailAddress
5        TextMessage number -> "Ok, i will text you at " <> number
6        Mail street town state zipcode -> "Ok, I will send a letter to " <> street <> "\n" 
7            <> town <> "\n" <> state <> "\n" <> show zipcode

This provides us with a data type that logically couples a bunch of different potential options that we can pass around our code without concern about what each sum type actually holds. It also allows us to store multiple different types in the same list

Haskell
1data StringOrNumber = S String | N Int
2
3stringsAndNumbers :: [StringOrNumber]
4stringsAndNumbers = 
5    [
6    S "this is a string",
7    N 2,
8    S "this is another string
9    ]

We can combine the record types we worked through earlier with sum types as well

Haskell
 1data CustomerInfo = CustomerInfo {
 2    customerName :: String,
 3    customerBalance :: Int
 4}
 5data EmployeeInfo = EmployeeInfo {
 6    employeeName :: String,
 7    employeeManagerName :: String,
 8    employeeSalary :: Int
 9}
10data Person =  Customer CustomerInfo | Employee EmployeeInfo
11
12chris = Customer $
13    CustomerInfo { customerName = "Chris Weaver",
14                   customerBalance = 100 }
15
16rachel = Employee $
17    EmployeeInfo { employeeName = "Rachel Weaver",
18                   employeeManagerName = "Tom",
19                   employeeSalary = 12 }

We are now ready to build out a function that can take the sum type Person and process its possible types and access the values within each type

Haskell
1getName :: Person -> String
2getName person =
3    case person of
4        Customer customer -> customerName customer
5        Employee employee -> employeeName employee
